# Havit 项目进度计划表

更新时间：2026-06-30

本文档用于回答三个问题：

1. 现在项目做到哪里了。
2. 功能应该怎么分区、怎么判断完成。
3. 接下来应该先向哪个方向发力。

进度判断依据：`docs/havit-product-design.md`、`docs/havit-tech-architecture.md`、当前前后端路由/API/服务/测试文件。百分比是产品可交付口径的估算，不是代码行数统计。

## 一、当前结论

项目已经不是概念阶段，也不只是骨架阶段。当前更准确的状态是：

| 层级 | 当前判断 | 完成度估算 | 说明 |
|---|---:|---:|---|
| M1 / P0 核心主链路 | 基本成型，进入验收与补洞阶段 | 85% | 物品、位置、认证、导入、分类、Docker、基础前端都已存在；主要缺完整验收、迁移清理和端到端覆盖 |
| M2 / P1 核心差异化 | 大面积铺开，部分能力待打磨 | 65% | 条码、AI、搜索、消耗品、EDC、二维码、借出、提醒、设置、多用户等都有实现痕迹；需要按真实场景收束 |
| M3 / P2 体验深化 | 已提前做了一部分，但不应继续扩张 | 40% | 虚拟资产、容器、理赔 PDF、PAT、EDC 打包等已出现；会话面板、确权变更、家庭内部借用、离线写仍未成型 |
| 发布信心 | 可以进入“内测稳定化”，不建议直接当正式版 | 58% | 模块很多，但验收、移动端一致性、隐私权限、异常路径和 E2E 还不够支撑正式发布 |

当前最重要的方向：先收束，不要继续横向加功能。优先把 P0 和高频日常链路做成可连续演示、可重复测试、可部署恢复的闭环。

## 二、状态标记

| 状态 | 含义 |
|---|---|
| 完成 | 数据模型、API、UI、权限、测试、文档都达到可验收状态 |
| 可用待验收 | 主流程已通，但测试、异常路径、移动端或权限仍需补齐 |
| 进行中 | 有部分模型/API/UI，但链路不完整 |
| 待开始 | 仅在产品文档中规划，代码中没有可靠实现 |
| 后置 | 明确不影响当前 MVP，可等稳定后再做 |

## 四、里程碑计划

### S0：基线整理与风险清理

建议窗口：2026-07-01 至 2026-07-03

| 目标 | 验收标准 | 产出 |
|---|---|---|
| 建立可重复验证基线 | `go test ./...`、`pnpm build`、现有 Playwright 全部通过 | 一份当前失败项清单 |
| 清理迁移风险 | `internal/db/migrations/0001_init.sql` 的 `Down` 段确认无误 | 迁移审计结论或修复 PR |
| 明确功能状态 | 按本文档表格更新每个模块状态 | 当前版本基线记录 |
| 统一术语 | UI 中产品命名与数据库标识映射一致 | 类型命名检查清单 |

优先级最高的风险：迁移文件当前在 `-- +goose Down` 后仍出现旧版建表内容，必须确认这是有意兼容还是历史残留。生产前不能带着不清楚的迁移语义继续叠功能。

|---|---|---|
| clean install | 空库 release 启动 -> `/setup` -> Owner 创建 -> 登录 | 后端单测 + Playwright |

| 导入导出 | CSV/JSON 导入后可搜索、可导出 | 单测 + Playwright |

| Docker | 前端构建嵌入后 Docker 启动可访问 | 手动验收脚本 |

S1 完成后，项目可以称为“MVP 可内测”。

## 五、当前最该发力的 10 件事

按顺序做，不建议跳过。

| 排名 | 任务 | 原因 | 完成判定 |
|---:|---|---|---|
| 1 | 跑通全量测试与构建基线 | 没有基线就无法判断进度涨跌 | `go test ./...`、`pnpm build`、现有 E2E 通过 |
| 2 | 审计并修正数据库迁移文件 | 迁移错误会影响后续所有版本 | clean DB 和迁移回滚语义明确 |
| 3 | 为 P0 主链路补 Playwright | 目前 E2E 太少，功能多但不可控 | setup、登录、资产、位置、导入、归档均有 E2E |
| 4 | 做一遍真实 Docker 部署验收 | 自部署是产品核心，不是附属项 | 新容器可启动、可 setup、可持久化数据 |
| 5 | 收束移动端高频页面 | PWA 用户主要在手机上查找和录入 | 资产、位置、详情、消耗品移动端可用 |
| 6 | 隐私和权限逐接口验收 | 家庭多用户承诺不能靠字段存在来证明 | 搜索、列表、位置树、导出、详情均不可越权 |
| 7 | 录入链路降级策略落地 | AI/条码不稳定，必须有人工兜底 | 拍照失败/扫码失败仍可手动保存 |
| 8 | 提醒与通知真实联调 | 借出、保修、库存价值依赖提醒闭环 | 可用 Webhook/Ntfy 收到业务提醒 |
| 9 | 备份恢复演练 | 自部署用户最怕数据丢 | 备份包可在新数据目录恢复 |
| 10 | 更新 README 的真实阶段 | README 仍停留在 M1 第二轮描述 | README 与当前项目阶段一致 |

## 六、完成定义

一个功能不能只因为“有表、有 API、有页面”就算完成。建议以后统一使用下面的完成定义：

| 检查项 | 要求 |
|---|---|
| 数据模型 | 字段完整，迁移可重复执行，索引与约束明确 |
| 后端 API | 正常路径、错误路径、权限边界都有测试 |
| 前端 UI | 桌面端和目标移动端都能完成主流程 |
| 离线/网络 | 网络断开时写操作有明确禁用或错误提示 |
| 隐私权限 | Member、Owner、私有资产、私有位置均有验收 |
| 可观测错误 | 用户知道失败原因，开发者能定位问题 |
| 文档 | README 或对应文档说明如何使用和验收 |
| E2E | 至少覆盖一条真实用户路径 |

只有以上全部满足，状态才从“可用待验收”改为“完成”。

## 七、每周进度控制节奏

每周固定做一次 30 分钟项目盘点，按下面顺序更新本文档：

| 步骤 | 内容 | 输出 |
|---|---|---|
| 1 | 跑基线命令 | 记录通过/失败 |
| 2 | 更新功能表状态 | 只在验收通过后上调完成度 |
| 3 | 检查上周新增风险 | 进入风险表，不在脑子里记 |
| 4 | 选本周最多 3 个目标 | 防止同时推进太多模块 |
| 5 | 为每个目标写验收标准 | 没有验收标准的目标不开始 |

建议每周目标数量上限：

| 类型 | 上限 |
|---|---:|
| 主功能收束 | 1 个 |
| E2E/测试补齐 | 1 个 |
| 风险/债务处理 | 1 个 |

## 九、建议的当前版本定位

当前版本建议称为：

> M2 收束前夜：P0 基本完成，P1/P2 多模块已铺开，正在转入验收、测试和产品化阶段。

不建议称为：

- 正式版
- P1 完成
- M3 开发期

原因是：高级功能已经出现，但主链路和验收体系还没稳。下一阶段的成功不取决于再加多少功能，而取决于能否让现有功能在真实家庭资产管理场景里连续、可靠、可恢复地跑起来。
