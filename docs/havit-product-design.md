# Havit — 产品设计规划

> 个人与家庭的全资产台账。实体的、虚拟的、消耗品的，都在一个地方。

**版本** v0.5 · 草稿  
**状态** 概念设计阶段

---

## 一、产品定位

### 一句话描述

Havit 是一款可自部署的家庭资产管理工具，帮助个人和家庭记录、追踪、查找所有"你拥有的东西"——无论是相机镜头、游戏机、食品库存，还是买断制软件授权。

### 目标用户

有一定数码/收藏品资产积累的个人用户或小家庭，对数据隐私有意识，愿意自己部署服务。典型画像：

- 拥有摄影器材、NAS、游戏主机等中高价值电子设备
- 家里有储量可观的食品、药品、日用品库存
- 有收藏习惯（手办、书籍、实体游戏）
- 购买过多款买断制软件/游戏，记不清楚自己都有什么

### 核心价值主张

| 现有产品的问题 | Havit 的解法 |
|---|---|
| 录入太麻烦，坚持不下去 | AI 拍照识别 + 条码扫描，一张照片完成录入 |
| 消耗品库存要手动减，没人维护 | 按消耗类型分策略：事件推算或一键计数，不强迫维护 |
| 只能管实体，买断软件/游戏没地方记 | 统一管理实体资产与虚拟买断资产 |
| 搜索靠关键词，找东西靠记忆 | 自然语言搜索 + 多维标签 |
| 随身物品根本不知道在哪 | EDC 双轨模型：基准位置 + 动态状态，搜索智能降级提示 |
| 到了现场还是找不到箱子 | 位置/容器可打印二维码，扫码反向知道里面装了什么 |
| 借出去的东西就找不回来了 | 内置借出追踪，到期提醒，含责任交割记录 |
| 保修单、发票到处找 | 凭证与保修信息与物品绑定，一处查询 |
| 东西丢了/卖了没有记录 | 全生命周期状态机：买入 → 使用 → 流转 → 消失，无死角闭环 |

## 二、资产类型体系

### 2.1 实体资产

**价格字段说明（全局适用）**

所有涉及金额的字段（购入价、二手售出价、损耗价值）均以 `金额 + 币种` 的格式原样存储（例如：`5000 JPY`、`299 CNY`、`49.99 USD`）。系统不做自动汇率换算，直接原样展示，保证数据干净、为未来扩展留后路。面向海淘、跨区购买的目标用户，这是基础的数据卫生要求。

---

**耐用品**（长期持有，不消耗）

记录重点：位置、保修期、序列号、购买凭证、估值（含币种）、资产状态。典型品类：

- 数码电子：相机镜头、游戏机、NAS 配件、耳机、电脑外设
- 收藏品：手办、实体唱片、书籍、实体游戏碟
- 家居器具：小家电、工具

**消耗品**（按消耗特性分为两种子类型，见 3.2）

- 类型 A — 高频流动性消耗品：纸巾、调料、零食、日用品等。去化快、连续消耗、成批购买。
- 类型 B — 低频长周期消耗品：净水器滤芯、打印机墨盒、空调遥控器电池、防潮箱干燥剂等。更换周期长、备件数量通常 ≤ 3 个、断货影响极大。

**EDC 物品**（随身携带、位置高频变动）

蓝牙耳机、钥匙、打火机、Kindle、充电宝等。采用"基准位置 + 动态状态"双轨模型（见 3.5），不要求实时维护位置。

**容器型物品**（既是资产，也可作为位置容器）

摄影包、工具箱、收纳盒等。在系统中同时扮演"资产"和"位置节点"双重角色，移动时内部物品路径自动随之更新（P2 功能）。支持生成可打印二维码（见 3.4）。

### 2.2 虚拟资产（买断制）

仅收录**一次性付费购入**的数字内容，不含订阅制服务（明确排除）。

记录重点：平台、订单号、授权邮箱/账号、购买日期、价格（含币种）、增补购买记录。典型品类：

- 买断游戏（Steam、GOG、PS Store、Nintendo eShop 等）
- 买断软件（Affinity、Sketch、Setapp 买断等）
- 电子书、数字音乐、影视的永久购买

**多平台凭证**：同一份内容跨平台重复购买时，支持在同一条资产记录下挂载多个平台凭证，避免重复条目。

**增补购买事件**：支持在主资产下追加大版本升级费、大型 DLC 等附加支出，清楚呈现对某一资产的总投入。

## 三、核心功能模块

### 3.1 录入系统

录入是 Havit 最重要的体验。目标：**录入一件物品的时间控制在 30 秒以内**。

**三种录入方式（按优先级）：**

① AI 拍照识别

拍一张物品照片 → AI 返回结构化识别结果 → 用户确认或修改 → 补充位置信息 → 保存。

**AI 幻觉防御机制**：多模态模型面对垂直领域物品（数码产品、收藏品）容易识别失准。为此：

- **结构化 Prompt 约束**：AI 识别结果必须严格映射到 Havit 的字段结构，AI 无法确定的字段（如精确型号后缀、序列号）必须留空，不允许模型填入推测值。用户在确认界面一键微调，而不是发现错误后反复修改。
- **原始照片视觉存证**：AI 识别所用的原始照片永久作为物品的封面图和视觉存证挂在资产上。即使 AI 把名字猜错了、用户忘了改，以后看一眼照片也能瞬间认出来。这张图片不可被自动覆盖，只能由用户主动替换。

② 条码 / QR 码扫描

扫描商品条码 → 从开放数据库（Open Food Facts、Barcode Lookup 等）自动填充商品名、品牌、图片 → 用户确认位置和分类 → 保存。降级策略：查询失败时 → AI 图像识别 → 若仍失败则呈现手动填写界面，不阻塞录入流程。

③ 手动快速录入

最小录入字段仅需：名称 + 位置（必填），其余字段全部可选，支持后续补充。

**批量导入（P0 功能）：**

支持 CSV / JSON 格式批量导入，提供字段映射向导，方便从 Notion、Excel、HomeBox 等工具平滑迁移已有数据。这是冷启动体验的关键，与数据结构同期交付。

---

### 3.2 消耗品追踪

消耗品按特性分为两种子类型，采用不同的追踪策略。

#### 3.2.1 类型 A — 高频流动性消耗品（购买事件模型）

放弃"实时库存数量"模式，改用**购买事件**驱动，辅以**轻量校准机制**。

**数据模型：**

每条记录包含两类事件：

- 购买事件：日期、购买数量、价格（含币种，可选）、备注
- 校准事件：用户主动触发的轻量修正信号

系统根据历史购买事件的时间间隔中位数推算"预计下次补货时间"，进入预警窗口后触发通知。

**用户操作流程：**

购物回来 → 扫描/拍照快速记录"买了什么、买了多少" → 系统更新购买事件 → 无需手动减库存

**校准机制：**

消耗速度因季节、生活变化而波动。用户无需修改任何数字，只需点击"快没了"或"还有很多"，系统将该信号记录为校准事件，重新修正后续预测曲线，维持提醒的可信度。

#### 3.2.2 类型 B — 低频长周期消耗品（一键计数模型）

消耗周期极长、备件数量极少的物品，历史样本量不足以支撑推算算法，改为**超轻量绝对数量追踪**。

**核心交互：**

物品卡片上直接显示当前备件数量（大字展示）。核心操作只有一个按钮："使用一个 / 已更换"，点击后数量 -1，全程不超过 2 秒。

**阈值报警：**

录入时设定最低库存阈值（默认为 1）。当数量低于阈值时，立即加入"待补货清单"并触发通知，无需推算耗尽时间。

**寿命倒计时（可选）：**

对于有额定寿命的物品（如"6 个月寿命滤芯"），在点击"已更换"的同时，系统自动为当前正在使用的那件创建到期提醒任务，复用保修提醒模块的同一机制。

### 3.3 搜索与定位

**自然语言搜索**

支持口语化查询：

- "我的备用 HDMI 线在哪"
- "有没有买过 Adobe 的东西"
- "快过期的药品"
- "借给小王的东西"
- "家里有哪些闲置超过半年的数码产品"
- "今年丢失了哪些东西"

搜索结果展示物品名称、完整位置路径、标签、状态、缩略图。处于异常状态（借出、EDC、丢失等）的物品，搜索结果附加对应的状态提示，而不是只展示基准位置。

**标签系统**

位置描述"在哪里"，标签描述"是什么属性"，两者互补。同一件物品只有一个当前位置，但可以拥有多个标签。例如：防潮箱中的镜头，位置是 `书房 → 防潮箱`，标签可以是 `#摄影 #尼康 #高频使用`。标签支持用户自由创建，系统根据分类和历史给出建议。

**位置路径显示**

所有物品显示完整位置路径，如：`卧室 → 床头柜 → 第二层抽屉`。EDC 物品和借出物品展示动态状态而非静态路径（见 3.5）。

---

### 3.4 空间与位置管理

采用**树形层级结构**，灵活适应各种家庭空间：

```
家（根节点）
├── 客厅
│   ├── 电视柜
│   └── 书架（第1层 / 第2层 / 第3层）
├── 卧室
│   ├── 衣柜
│   └── 床头柜
├── 厨房
│   ├── 冰箱
│   └── 橱柜
└── 储藏室
    ├── 货架 A
    └── 货架 B
```

系统内置以 `@` 为前缀的**动态虚拟节点**，与物理位置节点并列（见 3.5）：

```
@ 动态节点
├── @随身 (EDC)
├── @通勤包
└── @出差中
```

**轻量化物理标定（位置二维码）：**

文字路径告诉用户"东西在储藏室 → 货架 B → 收纳盒 3"，但到了现场面对一排外观相同的收纳盒，用户依然会犯难。为此，系统支持为任意位置节点或容器型物品生成**唯一编号 + 可打印二维码**（例如：`LOC-0042`）。

- **正向寻址**：搜索物品时，结果页附带该物品所在位置的编号标签，用户直接找对应标签的盒子。
- **反向寻址**：用户在现场扫描某个收纳盒上的二维码，系统立即弹窗展示该容器内的物品清单——比翻箱倒柜更快。

二维码打印页面内置于系统，支持批量生成后整页打印，无需第三方工具。

位置节点支持自由命名，层级深度不限。容器型物品（P2）可作为叶节点挂载到位置树，移动时内部物品路径自动同步更新。

---

### 3.5 EDC 与动态位置管理

针对蓝牙耳机、钥匙、打火机、Kindle 等高频变动、随身携带的物品，Havit 采用**"基准位置 + 动态状态"双轨模型**，拒绝要求用户实时维护位置。

**EDC 标记：**

物品录入时可勾选"常备随身（EDC）"属性。

**数据模型：**

| 字段 | 说明 |
|---|---|
| 基准位置 (Home Base) | 物品"该回的地方"，如 `玄关 → 置物盘` |
| 当前状态 (Current Status) | 具体物理位置，或动态节点（如 `@随身`） |

**搜索降级提示：**

当用户搜索一件 EDC 物品时，系统不生硬地返回基准位置，而是给出结构化提示：

> 💡 **当前状态**：该物品已标记为 `@随身携带`  
> 🏡 **如果不在身上**，请检查它的基准归宿：`玄关 → 置物盘`  
> 📅 **最后确认**：3 天前

**一键打包 / 归位（P2）：**

生成"出门检查清单"，列出所有 EDC 标记的物品，用户核对后一键将状态切换为 `@随身` 或 `@出差包`；回家后一键"全部归位"恢复基准位置。

---

### 3.6 凭证与保修管理

每件耐用品可附加：

- 购买凭证：照片或 PDF（发票、订单截图）
- 保修信息：购买日期、保修期限（自动计算到期日）、售后联系方式
- 序列号 / IMEI / SN
- 购买平台和订单号

**保修到期提醒**：在保修到期前 30 天、7 天各通过通知网关发送提醒。

**保修状态聚合视图**：可查看所有在保修期内及即将到期的物品列表。

---

### 3.7 借出追踪

**借出记录字段**：物品、借给谁（姓名/联系方式）、借出时间、预计归还时间、备注。

**提醒机制**：超过预计归还时间后通过通知网关发送提醒；用户手动标记"已归还"。

**借出状态**：物品标记为"借出中"，在物品详情和搜索结果中显示借出信息。

**责任交割**：若借出方将物品损毁或丢失，可在借出记录中标记"对方已弄丢"，记录赔偿情况（金额含币种、备注），流转至异常状态（见 3.9）。记录永久挂在物品历史日志中。

---

### 3.8 资产退场与流转

**正常退场状态：**

| 状态 | 说明 |
|---|---|
| 闲置 | 长期未使用，保留但不活跃 |
| 计划出售 | 标记为待出手 |
| 已售出 | 记录二手售出价（含币种）和售出时间 |
| 已赠出 | 转赠他人，记录接收方（可选） |
| 已报废/丢弃 | 损坏或主动丢弃 |

**闲置激活查询**：自然语言搜索支持"家里有哪些闲置超过半年的数码产品"，方便发现可出手或赠送的物品。

**流转台账**：物品退场时记录退场方式、日期、二手售出价（含币种）、备注。不做财务报表，但保留完整流转历史，供用户回溯"这件东西最后去了哪里"。

---

### 3.9 异常资产与损耗管理

物品并非只有正常退场。系统为丢失、被盗、损坏提供完善的异常状态和处理流程。**系统不允许直接删除物品**，统一通过状态流转归档，保留历史凭证。

**异常状态：**

| 状态 | 说明 |
|---|---|
| 已丢失 | 带出门遗失、记不清放哪了 |
| 被盗 | 失窃，需保留凭证用于报案或理赔 |
| 出借遗失 | 借出后借用方丢失，责任已通过借出模块记录 |
| 已损坏/报废 | 摔碎、烧坏、无法使用 |

**针对不同场景的处理流程：**

**被盗**：物品冻结，详情页顶部高亮标注被盗日期。支持一键导出理赔凭证（PDF，包含名称、购买金额及币种、发票截图、序列号/IMEI），直接用于报案或保险理赔（P2 功能）。

**遗失**：标记"已丢失"后，系统提示"是否加入补购清单"。若确认，自动在待办模块生成"补购 XX"任务，并保留原物品参数作为参考。支持在备注中记录最后一次见到的时间和地点。

**出借遗失**：从借出追踪模块直接流转（见 3.7），赔偿记录随物品历史日志保留。

**物品墓地（历史归档）：**

所有退场物品（正常或异常）从主搜索和位置树中隐去，移入"历史归档"（别名：物品墓地）。归档物品不占用日常查找精力，但随时可检索查阅全部历史信息，保证资产记录的完整性和连续性。

**损耗记录列表：**

可查看指定时间范围内的损耗记录清单（丢失、被盗、损坏的物品列表及购入价格和币种），帮助复盘生活习惯。这是一个记录列表，不做价值汇总财务分析。

---

### 3.10 虚拟资产管理

与实体资产统一在同一个系统中，通过资产类型区分。

虚拟资产特有字段：

- 平台（Steam / GOG / App Store / 独立开发者官网 等）
- 授权绑定的账号/邮箱
- 订单号或授权码（加密存储）
- 购买日期与价格（含币种）
- 下载/激活链接（可选）
- 多平台凭证（支持挂载多条，适用于跨平台重复购买场景）
- 增补购买事件（大版本升级费、大型 DLC 等附加支出，含币种）

不支持订阅制服务管理（明确排除在产品范围外）。

## 四、用户体验原则

**轻录入，重查找。** 录入门槛要低到用户愿意在购物回来的时候顺手记一下。查找要快到用户在找东西时第一反应是打开 Havit 而不是到处翻。

**不强迫习惯。** 消耗品不要求手动减库存。EDC 物品不要求实时更新位置。系统应该适应用户的行为，而不是要求用户改变行为。

**可选的复杂度。** 保修信息、序列号、凭证永远是可选的。用户可以只记"名字+位置"，也可以把所有信息都填满，两种用法都应该有价值。

**预测要可信。** 消耗品推算结果如果频繁失准，用户会完全忽视提醒。类型细分和校准机制是维持可信度的关键。

**AI 辅助，人工兜底。** AI 识别是降低录入摩擦的手段，不是替代人工判断的机制。不确定的字段留空、原始照片永久留存，让用户始终保持对数据的掌控感。

**全生命周期，无死角。** 物品从录入到退场，每一种结局都有对应的处理流程。任何时候都不应该需要"删除物品"来解决问题。

**数据归用户所有。** 自部署，数据存本地，不依赖任何第三方云服务。AI 识别功能可接入用户自己配置的 API Key，或完全离线的本地模型。

---

## 五、已明确的设计决策

### 5.1 多用户 / 家庭账号

采用"**物理空间共享，逻辑归属独立**"模型，并引入明确的角色层级：

**角色定义：**

| 角色 | 说明 |
|---|---|
| 全局管理员 (Instance Owner) | 部署者，唯一账户。负责系统配置：AI API Key、通知网关、条码库 API 等。普通成员不可见也无需接触这些配置。 |
| 家庭成员 (Member) | 普通用户，可录入和管理物品，权限受空间可见性约束。 |

**空间可见性规则：**

- 家庭共用空间（客厅、厨房、储藏室等）：所有成员可见、可搜、可编辑。
- 个人私有空间：仅归属人可见（例如：计划中的惊喜礼物、个人私藏）。
- **隐式共享默认行为**：普通成员在公共物理节点下新建物品时，该物品默认为"全家可见"，无需手动设置权限，防止因不熟悉权限配置而造成数据孤岛。

### 5.2 移动端策略

**PWA 优先**，不做原生 App。PWA + Web API 已足够调用设备摄像头进行扫码和拍照，规避自部署应用上架、跨平台证书、自签 HTTPS 等麻烦。后续视社区需求考虑原生封装。

### 5.3 AI 识别方案

双轨并行，用户自选：

- 云端模式：接入用户自己配置的 OpenAI / Anthropic / 其他兼容 API Key
- 本地模式：接入本地部署的多模态模型（如 LLaVA、Ollama），完全离线

### 5.4 条码数据库降级策略

优先查询 Open Food Facts（食品）和通用条码库。查询失败时降级为 AI 图像识别，若仍失败则呈现手动填写界面，不阻塞录入流程。

### 5.5 提醒系统

排除中心化邮件/短信推送，完全拥抱去中心化通知网关，由用户自行配置：Apprise（支持数十种通知渠道）、Bark（iOS）、Ntfy（跨平台）、Webhook（接入 Home Assistant、n8n 等）、Matrix / Telegram Bot。

### 5.6 数据备份与导出

内置定时自动备份（JSON 格式），备份文件存储在用户指定路径。同时支持随时手动导出为 CSV / JSON。

## 九、设置页规划

设置页采用双轨制：实例级配置仅 Owner 可见，用户级配置每位成员独立管理，互不干扰。

---

### 9.1 实例级全局配置（Owner 专属）

仅拥有 `owner` 权限的超级管理员可见和操作。配置存储于数据库 `system_configs` 表，修改后实时生效，**无需重启容器**。被宿主机环境变量锁死的配置项在 UI 中以只读状态展示，并附注说明。

**AI 与智能解析**

控制自然语言搜索和拍照识别的核心能力：

- 大脑选型：云端（OpenAI / Anthropic / Gemini）或本地（Ollama / LLaVA）切换
- API Key 与自定义代理地址（Base URL）
- 识别置信度策略：严格模式（不确定字段强制留空）/ 宽松模式（允许 AI 全字段猜测）——对应技术层的幻觉防御强度

**通知网关**

打通去中心化通知体系：

- 渠道激活：Bark（iOS）、Ntfy、Webhook、Telegram Bot
- Webhook 联动：可向 Home Assistant、Node-RED 推送资产状态变更事件
- 通知触发事件开关：消耗品低库存 / 借出逾期 / 保修到期 / 寿命到期，可按类型单独开关

**基础设施与媒体策略**

防范轻量硬件资源被击穿：

- 单张照片上传体积上限（默认 20MB）
- 缩略图 WebP 压缩质量滑块（画质与体积平衡）
- 成员注册开关：关闭后禁止新账号注册（公网暴露时防止被非法注册）

---

### 9.2 用户级个人偏好（全员可见）

每位成员独立配置，存储于 `user_preferences` 表，绑定个人 User ID，互不干扰。

**视觉调性与本地化**

- 主题：`亚麻台账 (Light)`（暖中性色调）/ `深夜书房 (Dark)`（黑胡桃木沉浸）/ 跟随系统
- 默认本位币种：录入物品时的默认货币符号（CNY / USD / JPY 等），不做汇率换算
- 日期显示格式：标准 `YYYY-MM-DD` 或相对时间（如"3 天前到期"）

**效率与工作流**

- 首页默认视图：进入 App 后优先展示哪个模块（空间物理树 / EDC 随身状态 / 待补货清单）
- 连续扫码行为：扫描条码后默认行为（打开 AI 确认页 / 静默入库并保持扫码流）

**隐私与个人通知**

- 新物品默认可见性：全家共享 / 个人私有（录入时的默认值，可单件覆盖）
- 个人通知设备 Token：绑定自己的 Bark Key 或 Ntfy 订阅，仅接收与本人直接相关的提醒（借出逾期、EDC 遗失），不受其他成员操作影响

---

### 9.3 物品墓地显示控制

已归档物品（丢失、被盗、已售出、已报废等所有退场状态）默认从主搜索和位置树中隐去，保持主界面清爽。

用户可在设置中开启"**在主搜索中显示已归档物品**"开关，开启后归档物品重新出现在搜索结果中（附加归档状态标记），适合进行年度资产盘点或理赔举证时使用。

此开关为**用户级个人偏好**，每位成员独立控制，不影响其他人的视图。
